LIGO — CTO Feasibility Questions

LIGO

This document describes what we want to build, stage by stage. For each stage we lay out the product vision, what we believe it requires, and the open questions we need a technical leader to answer: Is it feasible? How would you build it?

00 Technical Foundation

Before any single feature, we need foundational decisions that shape the entire product.

The App Itself

  • We want a mobile app that feels native on iOS and Android. What's the right way to build it — and what are the tradeoffs of each approach?
  • We need push notifications for daily questions, reveals, and matches. How would you implement that?

Data & Infrastructure

  • This app stores user profiles, music libraries, daily answers, match history, and chat messages. What database architecture would you use, and why?
  • We have time-critical, high-traffic moments (everyone opens the app at 8 PM for the reveal, Connection Night drops for the whole campus at once). How would you architect the system so it doesn't fall over at those moments?
  • We need background jobs — nightly reveal aggregation, music sync refreshes, Connection Night batch generation. How would you run and schedule those?
  • We need to store media — profile photos, album art, potentially audio snippets. What's your approach to file storage at scale?

01 Onboarding

What we want. A student signs up with their college ID and a few basics — age, gender, class year — and goes straight into the app. There is deliberately no music sync at this step; that moves to the profile. The experience should be fast and frictionless.
What we believe it requires. Account creation, session/auth, and storage of a minimal demographic set. A way to verify the person is a current student. A lightweight first-run that drops them into the daily experience quickly.

Open Questions

  • We only want real, current college students on the app. How do you verify that — and how do you do it without making signup feel like a government form?
  • We need to prevent duplicate and fake accounts. What's the best way to handle that at the account creation level?
  • Some freshmen are under 18. Does that change how we build the system from a compliance standpoint? What do we need to account for?
  • Students graduate. How should the system handle accounts when someone is no longer a current student? What happens to their data, their matches, their profile?
  • We have a Vibe (platonic) and Spark (romantic) matching system. That means we need to capture gender identity and orientation preferences at some point. How would you model that in the database, and how does it affect matching logic?

02 Home · Daily Moment

What we want. One music question appears the moment the user opens the app each day. They type and lock in an answer before that night's reveal. Every answer sharpens their taste profile over time. When they type, the input autocompletes from their own synced music library.
What we believe it requires. A daily-question content source and per-campus scheduling. Answer capture that is write-once and time-gated (locked before the reveal). A push notification to prompt the daily question. A fast, searchable copy of each user's music library for autocomplete.

Open Questions

  • We need a new music question every day for every campus. What's a sustainable way to source, schedule, and deliver those? Is it feasible to automate, or does this need a human content pipeline?
  • Once a user locks in their answer, it cannot be changed — and it must be locked before the reveal. How do you enforce that server-side, especially across time zones?
  • Each answer should feed into the user's taste profile. Is it better to update the profile in real time or batch it nightly? What are the tradeoffs?
  • What happens if a user doesn't answer on a given day? Should they still see the reveal? Can they still participate in Connection Night? We need to understand the technical implications of both paths.
  • We want the answer input to autocomplete from the user's own music library (their synced Spotify/Apple Music). Is it feasible to cache a user's library in our system and make it searchable in real time?

03 The Reveal

What we want. A nightly moment: the campus "sky" lights up with everyone's answers in a short, cinematic 5-act swipe-through sequence. It runs at a set time (e.g. 8:00 PM). Every student on campus sees the same reveal at the same time.
What we believe it requires. A nightly aggregation job per campus that assembles the reveal data. A rich client-side animation for the 5 acts. Push notifications and per-campus scheduling/timezone handling.

Open Questions

  • Potentially thousands of users will open the app at the exact same second. How do you architect the reveal so the system doesn't collapse under that simultaneous load?
  • We need to process and aggregate everyone's daily answers into a presentable reveal format. How would you build that pipeline? How far in advance does it need to run?
  • Should every user see the same reveal, or should it be personalized (e.g., "3 people who matched your answer")? What's the cost and complexity difference?
  • The reveal has to feel smooth and cinematic on any phone, including older models. Is a 5-act animated swipe sequence feasible on lower-end devices? What are the performance constraints?
  • What if only a small number of people answer on a given day? Is the reveal still compelling with a tiny dataset, or do we need a minimum threshold to trigger it?
  • We'd love to play a short audio snippet of the referenced song during the reveal. Is that possible? How would you source and serve those clips?
  • We'll eventually be on multiple campuses in different time zones. Does each campus get its own reveal time, or is there a smarter way to handle it?

04 Connection Night

What we want. Randomized roughly 1–2× a week, each user gets 3–5 match candidates. They can Vibe (platonic), Spark (romantic), or Pass on each one. Profiles are anonymous until mutual. A match only forms when intent aligns — Spark+Spark or Vibe+Vibe. A Vibe+Spark mismatch produces nothing, and neither person is told who chose them. The chosen person only sees "someone in your batch chose you."
What we believe it requires. A matching/batching engine that selects 3–5 candidates per user. A randomized scheduler with spacing rules (e.g. ≥48–72h between nights), per campus. Intent capture (vibe/spark/pass) and mutual-resolution logic. Anonymity guarantees in both the data layer and notifications.

Open Questions

  • We need a matching engine that selects 3–5 candidates per user based on music taste, demographics, and activity. How would you build that? What signals would you prioritize, and how do you score them?
  • Connection Night needs to happen 1–2× a week at random, with safe spacing. How would you schedule that per campus? What does the scheduler look like?
  • Anonymity is critical — no one should be able to figure out who chose them until it's mutual. How do you guarantee that technically? What are the leakage risks (notification timing, API responses, etc.)?
  • The Vibe+Spark mismatch silently produces nothing. How would you implement the mutual-resolution logic so it handles all intent combinations correctly?
  • At launch, the user pool will be small. If Georgetown has 200 users and you're filtering by gender preference, the eligible pool might be ~100. How do you prevent the same people from showing up over and over? What's the cooldown/recycle logic?
  • We need to track the full history of who was shown to whom, what they chose, and whether it was mutual. How would you model that data? What's the right structure for connections at scale?

05 Mutual Match → Messaging

What we want. When a match is mutual, a private chat opens with a soft ~7-day window. Because Connection Night happens 1–2× a week, match groups can overlap — all active mutuals live in a stories-style tray. On Georgetown's campus, people can nudge each other to events from the chat.
What we believe it requires. A real-time chat system. A message store with expiry/retention logic. The mutuals tray UI and push notifications for new messages.

Open Questions

  • We need real-time messaging between matched users. What's the fastest, most reliable way to build that? What approach would you take?
  • Chats have a soft 7-day window, but we need to retain messages for safety/abuse review even after they disappear from the UI. How would you handle the data lifecycle — what the user sees vs. what we retain, and for how long?
  • If someone reports a message on day 6, the system needs to preserve context for review. How does the reporting and moderation pipeline work from a technical standpoint?
  • Match groups overlap — a user might have 3 active chats from different Connection Nights at the same time. Does that create any scaling or data-modeling challenges?

06 Profile & Music Integration

What we want. Music sync lives on the profile as a mini-onboarding that earns a badge. Once synced, the app generates a live "Sonic Archetype" and a compatibility score between users. The public profile shows: photo, year, gender, clubs, a music block (On Repeat, Top Artists, a featured Playlist), and a "Why you're connected" panel. On your own profile, you see a private horoscope, your streak, connection count, and event RSVPs.
What we believe it requires. OAuth-based music service integration, data ingestion, and periodic refresh. Computation of the archetype and compatibility scores. A server-side view-resolver that swaps public/private fields depending on who's viewing.

Music Sync

  • We want users to connect their Spotify or Apple Music and have the app pull their listening data. Is that feasible with current APIs? What data can we actually get, and what can't we get anymore?
  • We want the synced music to power autocomplete when answering the daily question, plus populate "On Repeat" and "Top Artists" on the profile. How would you store and index that data so it's fast and stays fresh?
  • We want "On Repeat" to feel live — reflecting what the user is actually listening to right now. Can that be pulled automatically, or does the user have to curate it manually? What are the tradeoffs?
  • We want users to be able to feature a playlist on their profile. Is it possible to pull their playlists from Spotify/Apple Music and display the tracks? Can other users play it, or just see it?
  • Not every user will have Spotify or Apple Music. What's the experience for someone who doesn't connect a music service? Can they still participate fully?

Sonic Archetypes

  • We want to generate a "Sonic Archetype" for each user based on their music taste — something like "The Midnight Curator" or "Festival Circuit." Is there a technical approach (AI, clustering, rules engine) that can generate these reliably? What data would it need as input, and what are the key parameters?
  • We also want a compatibility score between any two users based on music taste. Now that Spotify has closed some of its advanced endpoints, what's still possible? How would you build a compatibility algorithm with the data that's available today?

Profile Privacy

  • The profile has public fields (photo, music, archetype) and private fields (horoscope, streak, connection stats). When one user views another's profile, private data should never reach their device — not hidden with CSS, actually never sent. How would you enforce that?

07 Audio Playback

Audio touches multiple features — the reveal, profiles, and potentially the daily question. It needs its own answer.

  • We want to play short audio snippets (15–30 seconds) of songs throughout the app — on profiles, during the reveal, maybe during the daily question. Is this possible? Where do the audio clips come from, and are there licensing or API constraints we need to know about?
  • Not every song may have an available preview. How do we handle that gracefully in the UI?
  • We want audio playback to feel seamless, not clunky. What's the best approach for playing short clips in a mobile app without requiring the user to have a paid streaming subscription?

08 "Your Artists This Week" — News & Events Feed

The Home screen has a horizontal news strip that shows artist-related content personalized to the user's music taste.

  • We want to show each user news and events related to the artists they listen to — new releases, concert announcements, music news. Is it possible to source this data programmatically? Is there an API for "music news by artist," or does this have to be built from scratch?
  • If we have to build it, how would you map artist-related news to individual users at scale? What does that pipeline look like, and how complex is it?
  • Is there a simpler V1 approach that gets us something useful at launch without building a full recommendation engine?

09 Trust, Safety & Moderation

⚠ Non-negotiable for launch. Any app where college students match, chat, and share profiles needs a safety and moderation layer built in from day one.
  • Users will upload profile photos. How do we screen for inappropriate content before a photo goes live?
  • Users will send messages in private chats. When someone reports a message, what should the technical pipeline look like — where do flagged messages go, how are they reviewed, and what's the response time?
  • We need block and unmatch functionality. When a user blocks someone, what should that actually do at the data layer? Does it prevent future matching, hide past conversations, or both?
  • Campus administration may contact us about student safety situations. What data are we able (and legally required) to share, and how do we build the system to support that?
  • Even after chats expire from the UI, we may need message history for safety investigations. How do we handle data retention so it serves both user privacy and safety obligations?
